- к каким бизнес-последствиям он может привести.

## Слайд 10. SmartDelivery: известные особенности

## Слайд 16. SmartDelivery: угрозы и последствия. Технический контур

**Формат:** таблица `угроза / недостаток защищенности -> где проявляется -> бизнес-последствие`.

**Представление:** сразу связывать угрозу с последствием. Не делать отдельный список угроз без объяснения, зачем они важны.

| Угроза / недостаток защищенности | Где проявляется | Возможные бизнес-последствия |
| --- | --- | --- |
| Внешняя атака на клиентский контур или API | Мобильное приложение, backend/API, оформление заказа | Клиенты не оформляют заказы; часть заказов не доходит до системы управления заказами; поддержка перегружена; теряется выручка в часы пик |
| Отсутствие единой аутентификации у части API | API между клиентским приложением, партнерами и внутренними сервисами | Пользователь или партнер получает доступ не к тем операциям; заказы могут быть изменены или отменены; расследование затягивается из-за разных механизмов доступа |
| Отказ платежного API | Платежная интеграция | Оплата не проходит или зависает; появляются двойные списания; финансовый отдел вручную сверяет операции; растут возвраты и претензии клиентов |
| Отказ картографического API | Приложение курьеров, маршрутизация, статусы доставки | Курьеры получают неверные маршруты; доставки опаздывают; растут компенсации клиентам; ухудшается выполнение SLA перед партнерами |
| Сбой облачного провайдера или облачного контура | Backend, API, административная панель | Приложение или внутренние сервисы недоступны; оформление и обработка заказов останавливаются; операционная команда переходит на ручной режим |

## Слайд 17. SmartDelivery: угрозы и последствия. Процессы, доступы, подрядчики

**Формат:** таблица `угроза / недостаток защищенности -> где проявляется -> бизнес-последствие`.

**Представление:** показать, что процессные и организационные недостатки защищенности дают такие же реальные бизнес-потери, как технические сбои.

| Угроза / недостаток защищенности | Где проявляется | Возможные бизнес-последствия |
| --- | --- | --- |
| Ошибка независимого изменения одного из модулей | Разные команды меняют заказ, оплату, авторизацию или интеграции независимо | Заказ оформлен в приложении, но не виден ресторану, курьеру или поддержке; растет число ручных корректировок и обращений |
| Старые модули не обновляются | Старые компоненты продолжают работать без обновлений и накапливают несовместимости с новыми сервисами | Появляются "потерянные" заказы; клиент, партнер и поддержка видят разные статусы оплаты или доставки |
| Злоупотребление расширенными доступами сотрудника | Административная панель, CRM, база заказов | Сотрудник просматривает лишние персональные данные, меняет статусы заказов или проводит ручные операции вне роли; возникают внутренние злоупотребления и финансовые потери |
| Компрометация учетной записи партнера | Веб-панель ресторана или магазина | Меняются меню, цены, статусы заказов или реквизиты; клиенты получают неверную информацию; возникают претензии по договору и финансовые споры |
| Недостаточный контроль подрядчиков | SDK, внешние интеграции, доступы к данным или API | Действия через интеграцию сложно отследить; при инциденте неясно, кто отвечал за изменение; интеграцию приходится останавливать до выяснения обстоятельств |

- трафик должен расти, но фактически падает.

## Слайд 27. QuickCart: события и сигналы для проверки

**Формат:** сравнительная таблица.
