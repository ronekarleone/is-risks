# Раскадровка урока 5 модуля 1

## Общая рамка

- Дисциплина: Управление рисками и процессами ИБ
- Модуль 1: Основы управления рисками информационной безопасности
- Урок 5: Контактное занятие. Идентификация активов и угроз информационной безопасности
- Формат: онлайн-семинар с демонстрационным разбором двух кейсов, короткими ответами в чат и самостоятельными паузами для записи
- Длительность: 90 минут
- Назначение раскадровки: опорный сценарий для разработки презентации и материалов семинара

## Центральная логика семинара

Семинар должен показать одну рабочую логику на двух разных ситуациях:

`контекст -> активы -> угрозы и недостатки защищенности -> последствия -> базовая карта рисков`

Кейс 1, SmartDelivery: плановая идентификация активов и угроз по описанию цифровой платформы доставки.

Кейс 2, QuickCart: событийный разбор проблемной ситуации после релиза.

## Рабочий ориентир по объему

- 34-37 слайдов
- 2 независимых кейса
- 2 рабочие таблицы
- 1 сравнительная таблица двух подходов
- 4-6 коротких пауз для ответа в чат или самостоятельной работы в записи

## Ожидаемый результат семинара

К концу занятия должны появиться два практических артефакта:

- базовая карта рисков SmartDelivery по архитектуре и зависимостям;
- карта риск-сценариев QuickCart по событийной ситуации.

Важно: это не полноценная количественная оценка рисков и не threat modeling. Это первый практический шаг: увидеть активы, угрозы, недостатки защищенности и последствия.

### Слайд 16. SmartDelivery: угрозы и последствия. Технический контур

- Функция слайда: сразу связать технические угрозы с бизнес-последствиями
- Основной тезис: угроза становится управленчески значимой только через конкретные последствия для заказов, денег, поддержки и SLA
- Что должно быть на слайде:
  - внешняя атака на API -> клиенты не оформляют заказы, заказы не доходят до системы, поддержка перегружена, теряется выручка в часы пик
  - отсутствие единой аутентификации API -> пользователь или партнер получает доступ не к тем операциям, заказы могут быть изменены или отменены
  - отказ платежного API -> зависшие или двойные платежи, ручная сверка финансов, возвраты, претензии клиентов
  - отказ картографического API -> курьеры получают неверные маршруты, доставки опаздывают, растут компенсации, нарушается SLA
  - сбой облака -> недоступны приложение, API или админка, оформление и обработка заказов останавливаются

### Слайд 34. Когда риск становится вопросом требований

- Основной тезис: требования нужны не как отдельная теория, а как ограничитель приоритизации; часть сценариев нельзя отложить только потому, что бизнес готов потерпеть неудобство
- Что должно быть на слайде:
